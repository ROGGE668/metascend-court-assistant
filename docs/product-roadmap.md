# Metascend 庭审助手 — 产品开发路线图

> 本路线图按“可交付、可测试、本地 macOS 优先”原则编排，目标是把项目推进到 **Phase 0 完成、Phase 1 稳定可测** 的状态，并逐步把前端对齐到现有 Web UI 设计稿。

## 关键里程碑

- **M1：可双击打开的 `.app`**
- **M2：离线全量测试通过**
- **M3：Phase 1 最小闭环稳定**
- **M4：前端与设计稿 1:1 对齐**
- **M5：内测准备就绪**

## Phase 0：基建与 Tauri 2 修复（第 1 周）

目标：先把“本地 macOS 案件工作台”的交付骨架修到可验证状态。

- 清理旧路线产物与过期文档描述。
- 修复 Tauri 2 构建/打包链路，确保能生成可直接打开的 `.app`。
- 统一 Web UI 设计稿与前端实现。
- 建立测试基线，保证后续每阶段可回归。

### 执行计划

| 时间 | 任务 | 验收 |
|------|------|------|
| Day 1 | 扫描 README、AGENTS、docs 中 court/mobile/cloud/端边协同描述并修正 | 关键词检索无过期叙事 |
| Day 2 | 修复 `frontend/src-tauri/tauri.conf.json`、Vite 构建输出、bundle 配置 | `npm run build` 稳定输出 |
| Day 3 | 修复 Tauri 2 capability/config 兼容问题 | `npx tauri build` 成功 |
| Day 4 | 手动验证 `.app` 可双击打开、状态灯/顶部栏/侧栏正常 | Finder 双击不报错 |
| Day 5 | 建立回归测试脚本与 checklist | 一键脚本可复验 |

## Phase 1：后端稳定与测试（第 2-3 周）

目标：在 Tauri 外壳稳定后，先保证 Python 后端在本地 MacBook 上稳定可测。

- 运行 `uv sync` 并完成模型预下载脚本验证。
- 完成音频采集、VAD、ASR 单测与联调。
- 完成 diarization、legal、TTS 模块的冒烟测试。
- 建立“模块完成 -> 单测 -> 联动测试”的固定节奏。

### 执行计划

| 时间 | 任务 | 验收 |
|------|------|------|

| Week 3 | diarization + legal 基础联调 | 2-4 人场景可打标 |
| Week 3 | 后端服务健康检查与本地模型缓存校验 | 断网可启动核心能力 |

- 延迟、准确率、稳定性压测。
- 隐私与本地化验证：断网、加密存储、权限最小化。

## 每阶段执行纪律

1. **每完成一个模块**：先跑单测，再做与已完成模块的联动测试。
2. **每阶段结束**：跑全量回归，更新 checklist，记录已知风险。
3. **问题修复顺序**：先修复阻断打包/启动的问题，再优化体验细节。
4. **测试优先**：UI/交互/打包问题先做手动验证，再补自动化回归。
5. **文档同步**：代码或架构有变化时，同步更新对应文档。

## 最大风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 法律建议错误 | 误导用户、合规风险 | 所有输出带免责声明；高风险建议强制过滤；律师复核 |
| Tauri 构建/打包失败 | 无法交付 `.app` | 每阶段优先保构建可运行；旧产物逐步清理 |
| 本地模型资源占用高 | MacBook 发热/卡顿 | 按需加载、独立进程、量化模型、降级策略 |
| 前端与设计稿不一致 | 产品体验差 | 先视觉对齐，再补交互细节 |
